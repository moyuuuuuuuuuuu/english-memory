# Flutter Reliable Sync Queue Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Persist every confirmed Capture draft before networking, then send card-creation requests through a restart-safe, account-isolated, idempotent queue.

**Architecture:** Drift owns the local card placeholders, creation outbox, and per-account sync cursor. A repository performs atomic local mutations, a Dio API adapter translates the existing backend envelope, and a single-flight coordinator serially drains retryable work. Capture and session UI depend on narrow gateways so persistence, networking, and logout cleanup remain independently testable.

**Tech Stack:** Flutter 3.44.7, Dart 3.12.2, Drift/SQLite, Dio, UUID v4, flutter_test.

## Global Constraints

- Work only on `codex/flutter-reliable-sync-queue`; do not merge or push until the user requests it.
- Use red-green-refactor for every behavior change and commit each completed task.
- Every local business row is scoped by the authenticated numeric `account_id`.
- `local_id` is UUID v4 and is the immutable `Idempotency-Key` for an unchanged request.
- Queue sends are serial; network/timeout/5xx/429 retry, 401 pauses without incrementing attempts, and other 4xx block.
- Retry delay is exponential from 2 seconds, capped at 5 minutes, with deterministic jitter injection for tests; valid `Retry-After` may extend it.
- Initial delivery runs only while the application is active. Startup, authenticated session restoration/login, foreground resume, enqueue, and manual retry trigger draining.
- Do not log Capture text, credentials, bearer headers, or raw error bodies.
- Do not implement full library sync, generation polling, background OS tasks, or offline edit/delete/review queues.
