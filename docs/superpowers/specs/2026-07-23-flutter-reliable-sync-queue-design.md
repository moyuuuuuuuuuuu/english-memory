# Flutter 本地缓存与可靠待同步队列设计

## 目标

为 Flutter 客户端建立可靠的本地持久化基础，使用户确认一条英文内容后，即使断网、请求超时、应用退出或进程被系统终止，也不会丢失待生成的记忆卡。

本阶段只实现记忆卡创建的本地缓存与可靠发送队列。完整卡片内容同步、生成进度轮询、卡片库界面和冲突处理界面属于阶段 8，不在本设计范围内。

## 已确认的产品决策

- 用户确认 Capture 内容时总是先写入本地数据库，再尝试请求后端。
- 在线和离线提交共用同一条队列路径。
- 本地写入成功后立即向用户反馈“已保存，正在同步”，不等待网络。
- 退出登录时，如果存在待同步内容，必须显示具体数量并请求确认。
- 用户确认退出后，删除该账号的缓存、同步游标和队列；取消退出则保留数据并继续尝试同步。
- 初版仅在应用前台运行队列，不承诺关闭应用后的系统后台上传；下次启动自动恢复。

## 技术选型

使用 Drift + SQLite 作为本地持久化层。

选择 Drift 的原因：

- 事务能够保证草稿、缓存卡片和队列项原子写入。
- 类型安全查询和显式迁移适合后续扩展卡片、标签、墓碑及同步版本。
- 响应式查询可为待同步数量和后续卡片库直接提供数据流。
- SQLite 的关系模型与后端卡片、标签、关系和用户级同步版本结构一致。

不选择 sqflite，因为它需要手写 SQL 映射和响应式更新；不选择 Hive CE，因为后续按同步版本查询并维护卡片与标签关系时会增加额外复杂度。

## 架构

### 本地数据库

本地数据库按 `account_id` 隔离所有业务数据。第一版包含三张表：

#### `pending_card_creations`

保存必须可靠发送的创建请求：

- `local_id`：客户端生成的 UUID，同时作为稳定的 `Idempotency-Key`
- `account_id`
- `text`
- `content_type`
- `memory_style`
- `state`：`pending`、`sending`、`retry_wait` 或 `blocked`
- `attempt_count`
- `next_attempt_at`
- `last_error_code`
- `created_at`
- `updated_at`

`local_id` 创建后永不改变。相同队列项的所有重试必须发送完全相同的请求体和幂等键。

#### `cached_cards`

保存本地可观察的卡片占位记录：

- `local_id`
- `account_id`
- `source_text`
- `content_type`
- `memory_style`
- `local_status`
- 可空的 `server_card_id`
- 可空的 `server_job_id`
- 可空的 `generation_status`
- `created_at`
- `updated_at`

后端接受请求后写入服务端 Card ID、Job ID 和 `queued` 生成状态。阶段 8 可通过数据库迁移扩展完整卡片内容、图片信息、内容版本和同步版本。

#### `sync_state`

每个账号保存一条同步状态：

- `account_id`
- `cursor`
- `updated_at`

本阶段只建立数据边界；阶段 8 接入 `GET /api/sync/changes` 时才推进游标。

### 组件边界

- `LocalAppDatabase`：定义表、迁移和底层事务。
- `CardOutboxRepository`：原子保存 Capture 草稿与缓存占位卡；查询并更新队列项；按账号统计和清理。
- `MemoryCardApi`：只负责将创建请求发送到 `POST /api/memory-cards`，解析统一响应信封。
- `CardSyncCoordinator`：串行提取队列项、调用 API、分类错误并更新本地状态。
- `CaptureController`：保持当前图片选择、裁剪、OCR 和编辑职责；提交时委托本地仓库，不直接访问网络或数据库实现。
- `SessionController` 或应用级会话编排：登录后启动对应账号的队列；退出前查询待同步数量；确认退出后清理账号数据。

组件通过接口注入，生产环境使用 Drift 和 Dio 实现，单元测试使用内存数据库或测试替身。

## 数据流

### 创建

1. 用户确认 Capture 内容。
2. 客户端生成 UUID。
3. 单个数据库事务同时插入 `cached_cards` 占位记录和 `pending_card_creations`。
4. 事务成功后清空 Capture 输入并显示“已保存，正在同步”。
5. 通知 `CardSyncCoordinator` 尝试排空当前账号的队列。

如果本地事务失败，不清空 Capture 内容，并显示本地保存失败。没有成功落盘的数据不得发送到后端。

### 发送成功

1. 协调器按 `created_at`、`local_id` 的稳定顺序选择最早一项。
2. 在本地将其标记为 `sending`。
3. 使用 `local_id` 作为 `Idempotency-Key`，请求体使用已持久化的字段。
4. 后端返回 HTTP 202 后，在单个事务中：
   - 将 `server_card_id`、`server_job_id` 和 `generation_status` 写入缓存卡片；
   - 将本地卡片状态更新为 `queued`；
   - 删除对应待发送项。
5. 继续发送下一项。

若后端已成功而客户端在本地事务完成前退出，下次会使用相同键和请求体重试。后端将稳定重放相同 Card ID 和 Job ID，本地随后完成映射，不会重复创建。

### 应用恢复

数据库打开后，将当前账号遗留的 `sending` 项恢复为 `pending`。以下事件触发一次排队发送：

- 登录或已保存会话恢复成功；
- 应用回到前台；
- 新草稿成功落盘；
- 用户对阻塞项执行重试。

网络连通状态只能用作触发提示，不能作为成功依据。只有 API 响应才能推进队列状态。

## 队列与重试策略

- 同一账号严格串行发送，初版不并发创建 AI 卡片。
- 网络异常、连接超时、HTTP 5xx 和 HTTP 429 属于暂时性错误。
- 暂时性错误进入 `retry_wait`，增加 `attempt_count`，并设置带随机抖动的指数退避时间。
- 建议退避基数为 2 秒，上限为 5 分钟；若 429 提供有效 `Retry-After`，使用两者中更晚的时间。
- HTTP 401 暂停队列且不增加尝试次数。由既有鉴权客户端尝试刷新；若会话最终失效，回到登录页并保留该账号本地数据，等待用户重新登录或明确退出。
- HTTP 422 和 409 属于永久性请求错误，进入 `blocked` 并保留内容和错误码。
- 未识别的 4xx 默认进入 `blocked`，避免无限重试无效请求。
- 协调器在任意时刻只允许一个 drain 执行；重复触发合并为同一轮或紧随其后的新一轮。

用户对 `blocked` 项选择“重新发送”且请求内容未变化时，可将原项恢复为 `pending`，继续使用原幂等键。若用户修改了内容，则删除或归档原失败项，并通过本地事务创建新卡片占位记录和新幂等键。

## 用户界面

- Capture 本地保存成功后显示短暂成功提示，并清空文本和已选图片。
- 页面提供可观察的待同步数量；暂时性失败不弹出阻断式对话框。
- 永久错误显示“需要处理”，允许用户查看原文并重试。
- 退出时若待同步数量大于零，提示：“还有 N 条内容尚未同步。退出后将删除这些内容和本机缓存。”
- 用户取消退出时不清理任何数据，并重新触发队列。

本阶段只提供支撑上述流程所需的最小入口，不实现完整 Library 卡片列表或生成进度详情。

## 数据安全与账号隔离

- 所有查询、更新和删除必须显式包含 `account_id`。
- 本地业务数据不写入 `flutter_secure_storage`；安全存储继续只保存会话凭证。
- 日志不得输出 Capture 原文、Bearer Token、Refresh Token 或完整后端错误体。
- 确认退出后，在撤销会话的同时清理对应账号的三张本地表数据。
- 另一个账号登录时不得读取或发送前一账号的队列。

## 测试策略

遵循红—绿—重构，先写失败测试。

### 数据层

- 首次建库和迁移。
- 草稿与占位卡原子写入。
- UUID 幂等键持久且稳定。
- 所有读取和清理均按账号隔离。
- 应用恢复时将遗留 `sending` 置回 `pending`。
- 退出确认后的账号级清理。

### API 与协调器

- 在线 HTTP 202 成功并保存 Card ID、Job ID。
- 相同幂等键重放后完成本地映射。
- 断网、超时、5xx 和 429 进入退避。
- 401 暂停但不增加尝试次数。
- 422、409 和其他不可恢复 4xx 进入 `blocked`。
- 指数退避不超过五分钟并支持 `Retry-After`。
- 队列严格串行，多个 drain 触发不会重复并发发送。
- 成功项删除后继续处理下一项。

### 展示与会话

- Capture 保存期间禁用重复提交。
- 保存成功后清空输入并显示反馈。
- 本地保存失败时保留输入。
- 待同步数量正确更新。
- 有待同步项时退出显示数量；取消退出保留数据；确认退出完成清理。
